# 023 -- URL-safe taxonomy term slugs (facet + term-page links)

## Context

Filed from the **libcatalog-demo** adopter site (uncommitted, per the workspace
convention -- a request, not in-progress). The demo hit a live 404 that traces to the
module.

`layouts/_partials/facets.html` and `layouts/page.html` build facet/subject/tag links
with `urlize <label>`, and Hugo mints the matching term pages by slugifying the taxonomy
term. `urlize` preserves some characters that are unsafe in a URL **path** on a real CDN.
Concretely, a controlled subject labeled `LGBTQ+ books` produces the link+page slug
`lgbtq+-books`; a browser sends the literal `+`, which S3/CloudFront decode as a space, so
`/subjects/lgbtq+-books/` returns **404** while only the `%2B`-encoded form resolves.
(Accented/dotted terms are fine -- browsers percent-encode non-ASCII, and dots are
path-safe; `+` is the sharp edge, and other ASCII punctuation like `&`, `#`, `?` would be
similar.)

The demo worked around it by dropping the `+` from its display label, but the fix belongs
here so every adopter's facets are click-safe regardless of authority labels.

## Proposed change

Slugify term links and term-page paths with a single URL-safe function, consistently on
both sides so they always match:

- Replace `urlize` in `facets.html` / `page.html` links with `anchorize` (or a helper
  that lowercases, maps runs of non-`[a-z0-9]` to a single `-`, and trims) so `+`, `&`,
  etc. never reach the path.
- Ensure the taxonomy term pages Hugo generates use the **same** slug (set an explicit
  term slug, or normalize the label the content adapter feeds into the taxonomy) so the
  link and the page agree.
- Add a test/fixture with a `+`-bearing label (e.g. a Homosaurus `LGBTQ+ ...` term) and
  assert the emitted link resolves to the generated term path.

## Acceptance

- A subject/tag whose label contains `+`, `&`, or similar renders a facet link that
  resolves to its term page (no 404, no reliance on manual `%`-encoding).
- Link slug and term-page path are generated by one shared function.

## Done (2026-07-02)

Shared slug partial `layouts/_partials/lcat-slug.html` (lowercase, collapse runs of
non-`\p{L}\p{Nd}` to one `-`, trim), used on **both** sides so link == term page:

- **Content adapter** (`_content.gotmpl`) feeds the *slug* as the `subjects` and `tags`
  taxonomy values (the free-text dims that carry `+`/`/`), so Hugo mints the term page at
  the safe path. Human labels are preserved for display via `subjectList` and a new
  `tagLabels` param. Contributors/formats/languages/classifications stay raw (already
  urlize-safe; slug of them is urlize-equivalent, so links still match).
- **Facet links** (`facets.html`, `page.html`) go through `partial "lcat-slug.html" | urlize`.
- Fixture: the exampleSite carries a `LGBTQ+ books` tag; before, Hugo minted
  `/tags/lgbtq+-books/` and the CDN would 404; now both link and page are
  `/tags/lgbtq-books/`.
- **Also fixed** (surfaced by the new link check): facet links used `relURL`, so on
  non-default-language pages a localized subject slug (e.g. es `personas-trans`) pointed
  at the missing default-language path and 404'd. Switched to `relLangURL`, so each
  language's facet links resolve within its own language tree.
- **Test:** `link_check.cjs` (+ `npm run test:links`) walks the built site and asserts
  every internal facet/term/work link resolves and no path contains a literal `+`.
  Verified it catches the bug (negative control) and passes on the fix; a11y still clean
  over all 95 pages.

### Known tradeoff (option b)

Because the `subjects`/`tags` taxonomy *value* is now the slug, those term pages' own
`<h1>` shows the slug-derived title (e.g. "Gender-Identity", "Lgbtq-Books") instead of the
human label. Detail pages, the facet sidebar, and contributor term pages keep pretty
labels. Restoring pretty subject/tag term titles would need explicit per-term slugs
(taxonomy content adapters) -- a possible follow-up if desired.
